# 架构设计与技术决策

## 项目概述
- **项目名称**：astrbot_plugin_web_analyzer
- **主要功能**：Web 内容分析，包含网页截图等功能
- **宿主环境**：AstrBot
- **主要语言**：Python

## 技术栈
- **核心依赖**：
  - `httpx`：网络请求（使用秒作为超时单位）
  - `playwright`：浏览器自动化（使用毫秒作为超时单位）
- **双客户端架构**：同时管理 HTTP 客户端和浏览器自动化

## 架构设计

### 分层架构
```
配置层 → 入口层 → 消息处理层 → 核心分析层
```

### 配置驱动设计
- **行为完全由配置决定**：如 `self.timeout` 从配置层透传到多处异步调用
- **灵活性高**：通过配置可以调整各种行为参数
- **测试复杂度增加**：需要测试多种配置组合

### 截图等待策略（策略模式）
支持三种模式并带有降级链机制：
1. **fixed**：固定等待时间
2. **networkidle**：等待网络空闲（在 SPA 页面可能永远无法触发）
3. **smart**：智能等待策略

**降级链机制**：当一种策略失败时，自动降级到下一种策略，确保截图功能可用性。

### 单文件核心倾向
- **当前设计**：`analyzer.py` 承载 HTTP 请求、截图、策略选择等多职责
- **潜在问题**：职责分散，存在耦合，变更时需警惕牵一发动全身
- **改进方向**：考虑职责分离，将不同功能模块化

## 关键设计决策

### 1. 超时配置统一管理
- **决策**：所有浏览器操作的超时必须统一使用实例超时配置
- **理由**：避免硬编码超时值，提高配置灵活性
- **实现**：`self.timeout` 从配置层透传到各调用点

### 2. 双客户端超时语义一致性
- **挑战**：playwright 使用毫秒，httpx 使用秒
- **决策**：在配置解析时进行单位转换，确保语义一致
- **风险**：单位转换错误可能导致超时设置不当

### 3. 防御性编程与优雅降级
- **决策**：采用超时不阻断的设计，遇到异常时执行优雅降级
- **理由**：确保核心功能可用性，提高用户体验
- **实现**：错误信息从笼统错误转向带原因提示

### 4. 参数构建收敛
- **决策**：浏览器启动参数（如 `--no-sandbox`）应统一在构建器中生成
- **理由**：避免参数散落在重试逻辑或初始化逻辑中
- **实现**：在 `_build_browser_launch_args` 等构建器中统一生成

### 5. 版本号同步机制
- **决策**：`main.py` 与 `metadata.yaml` 双版本号文件必须强制同步
- **理由**：避免版本不一致导致的问题
- **实现**：建议在 CI 中增加脚本校验

## 已知技术挑战

### 1. SPA 页面截图局限性
- **问题**：`networkidle` 策略在纯 SPA 页面可能永远无法触发
- **解决方案**：依赖 `smart` 策略的降级机制兜底
- **设计关注点**：降级触发条件是核心设计的关注点

### 2. httpx 版本陷阱
- **问题**：httpx 在 0.24 到 0.28 之间经历了 `proxies` (dict) 到 `proxy` (str) 的破坏性底层重构
- **风险**：最低版本硬性拉高到 0.28 可能会阻断旧宿主环境用户
- **应对**：审查依赖版本提升时，必须注明依赖的特定 API/Feature

### 3. 类型安全风险
- **问题**：配置值为字符串时（如 `"30"`），`self.timeout * 1000` 会引发类型错误
- **解决方案**：在配置解析时转换为数值类型，或增加运行时防御

### 4. 容器环境检测假阳性
- **问题**：解析系统命令输出时，依赖固定数量空白符匹配可能导致误判
- **解决方案**：使用 `split()` 或正则按语义提取，引入白名单机制

## 架构演进方向

### 1. 职责分离
- **现状**：`analyzer.py` 职责过重
- **改进**：考虑将 HTTP 客户端、浏览器自动化、策略选择等功能模块化

### 2. 配置验证增强
- **现状**：配置边界值未在代码或测试中明确处理
- **改进**：增加配置验证层，明确处理边界值（0、负数、None、空字符串）

### 3. 测试覆盖完善
- **现状**：超时、重试、降级逻辑的测试覆盖不足
- **改进**：要求涉及这些功能的 PR 必须附带或说明测试覆盖情况

### 4. 错误处理标准化
- **现状**：异常包装不一致，可能损害可观测性
- **改进**：建立异常处理规范，禁止使用无上下文的笼统错误

## 审查与质量保障

### 审查重点
1. **配置同步**：配置项变更必须同步更新 `_conf_schema.json`
2. **超时一致性**：所有浏览器操作的超时必须统一使用实例超时配置
3. **版本管理**：版本号变更时检查所有版本定义文件
4. **重构等价性**：重构方法必须确认行为绝对等价

### 质量保障措施
1. **CI 校验**：建议在 CI 中增加版本号同步校验脚本
2. **测试要求**：涉及超时、重试、降级逻辑的 PR 必须附带测试
3. **文档更新**：新增配置项必须考虑旧配置文件的兼容处理
