# 审查知识总结

## 从反思中提取的关键模式

### 1. 审查质量评估维度

#### 覆盖度不足的常见表现
- 遗漏关键文件检查（如 `_conf_schema.json`、`CHANGELOG`、`README`）
- 未验证边界条件（如 `networkidle` 超时阈值合理性）
- 未检查向后兼容性（旧配置文件升级时的默认值）
- 未验证测试覆盖情况

#### 准确度偏颇的常见表现
- "光环效应"：全部是正面建议，零问题发现
- 复杂度不同的变更一概而论
- 评分与变更复杂度不匹配

#### 完整性表面化的常见表现
- 声称"参数透传完整"但未展示验证路径
- 未检查关键路径的可观测性（如策略降级日志）
- 未验证重构等价性

### 2. 评分校准规则（严格执行）

| 变更类型 | 建议评分 | 说明 |
|----------|----------|------|
| 纯版本号/注释/格式化变更 | ≤ 8/10 | 不应超过 8/10 |
| 功能完成但未验证边界、测试、文档 | 7-8/10 | 从 9/10 降至 7-8/10 |
| 包含重构但未验证等价性 | ≤ 8/10 | 不应超过 8/10 |
| 功能完成 + 测试覆盖 + 无任何建议项 | 10/10 | 仅用于完美情况 |

### 3. 代码审查关键规则

#### 配置相关规则
- **schema-sync**：配置项变更必须同步 `_conf_schema.json`
- **backwards-compat**：新增配置项必须考虑旧配置文件兼容处理
- **config-boundary-test**：配置边界值必须在代码或测试中明确处理
- **timeout-type-safety**：使用 `self.timeout` 进行算术运算时须确保类型安全

#### 超时与重试规则
- **screenshot-timeout-consistency**：浏览器操作超时须统一使用实例配置
- **pr-test-requirement**：涉及超时、重试、降级逻辑的 PR 必须附带测试
- **strategy-logging**：策略降级必须记录日志（含原因和降级目标）

#### 版本管理规则
- **version-bump-completeness**：版本号变更时检查所有定义版本号的文件
- **patch-version-context**：patch 版本号递增时需确认对应 bug 修复已合入
- **dep-version-context**：依赖最低版本提升必须注明依赖的特定 API/Feature

### 4. 常见问题模式

#### 配置相关问题
1. **类型安全盲区**：配置值为字符串时参与算术运算
2. **配置边界未验证**：`timeout=0`、负数、`None`、空字符串未处理
3. **配置同步遗漏**：配置项变更未同步更新 `_conf_schema.json`

#### 超时与重试问题
1. **硬编码超时反模式**：浏览器操作中使用硬编码超时值
2. **双客户端超时语义不一致**：playwright 使用毫秒，httpx 使用秒
3. **策略降级可观测性不足**：降级时未记录日志或日志信息不完整

#### 版本管理问题
1. **分离式版本号提交**：核心修复与版本递增拆分为独立 PR
2. **版本号同步遗漏**：双版本号文件（`main.py` + `metadata.yaml`）不同步
3. **依赖版本提升风险**：硬性拉高最低版本可能阻断旧宿主环境用户

### 5. 生命周期管理新模式

#### 功能对称性审查
- **核心原则**：当插件提供"安装/初始化"功能时，应主动检查是否存在对应的"卸载/清理"能力
- **审查要点**：安装→卸载、初始化→重置、启动→停止、启用→禁用

#### 删除操作前置条件
1. **进程占用检查**：检查浏览器是否正在被其他进程使用
2. **权限降级处理**：处理 `shutil.rmtree` 的权限问题
3. **失败回滚机制**：删除失败时不应导致插件崩溃

#### 异步状态同步
- **竞态条件风险**：`_is_installing` 与浏览器池忙碌状态的复合检查
- **解决方案**：使用 `asyncio.Lock` 保护状态检查，状态持久化到配置文件

### 6. 审查流程优化

#### 增量审查注意事项
1. **强制检查历史上下文**：审查版本递增等孤立提交时，应主动调取历史修复内容
2. **避免修复遮蔽效应**：警惕视线被锚定，忽略同一文件中其他函数的变更
3. **阅读核心入口文件**：无历史记忆的增量审查若涉及新功能，须阅读核心入口文件

#### PR描述与实际变更对照
- **规则**：PR描述中提到的文件必须出现在diff中，否则需追问原因
- **目的**：避免遗漏重要变更或存在误导

### 7. 仓库特有规范

#### 双客户端架构注意事项
- **超时语义一致性**：playwright 使用毫秒，httpx 使用秒，审查时须留意单位转换
- **配置透传模式**：超时、重试等参数从配置层 → 构造函数 → 多处异步调用

#### 截图功能审查要点
- **策略降级链**：`fixed`、`networkidle`、`smart` 三种模式的降级触发条件
- **SPA 页面局限性**：`networkidle` 策略在纯 SPA 页面可能永远无法触发
- **错误信息可观测性**：异常包装必须传递原始异常信息

#### 版本号同步要求
- **双版本号文件**：`main.py` 与 `metadata.yaml` 必须强制同步
- **版本号提交应与变更合并**：建议核心修复合入时立即更新版本号

### 8. 经验教训总结

#### 值得持续关注的要点
1. **类型安全盲区**：Python 动态类型下，配置值的运行时类型假设是高发风险点
2. **配置默认值的可观测性**：当配置值来自外部时，审查应验证默认值是否合理且被正确传递
3. **评分校准**：避免"改动小 = 高分"的心理偏差，功能正确性 ≠ 完备性

#### 技术审查经验
- **httpx API 变更**：`proxies` → `proxy` 是破坏性变更，需确认最低版本要求
- **容器环境检测**：Docker/K8s/LXC/WSL/systemd-nspawn 各有特征，检测逻辑需要持续维护
- **Linux uid_map 格式**：列间空格数量是内核实现细节，不应作为匹配依据

#### 审查流程优化
- **强制检查历史上下文**：避免增量审查的"盲区"
- **阅读核心入口文件**：无历史记忆的增量审查若涉及新功能，须阅读核心入口文件
- **风险缓解建议**：对于删除操作，应提供失败回滚机制或手动删除指引
