# 经验教训与常见问题模式

## 审查经验教训

### 评分与决策
- **高评分陷阱**：改动小 ≠ 高分。功能正确但未验证边界、测试、文档时，评分应从 9/10 降至 7-8/10
- **纯版本号提交**：评分不应超过 8/10，10/10 仅用于功能完成 + 测试覆盖 + 无任何建议项
- **重构等价性**：包含重构但未验证等价性的 PR，评分不应超过 8/10
- **决策必须明确**：禁止使用 `unknown`，强制选择 `approve` 或 `request_changes`

### 增量审查盲区
- **小步迭代提交**：Bug 修复和新功能常拆分为多次提交，最后用单独提交修正版本约束和日志细节
- **增量审查易丢失全局视角**：仅看 diff 无法评估新功能在入口文件中的调度逻辑是否正确
- **修复遮蔽效应**：当上一轮遗留问题集中在特定文件时，审查员视线容易被锚定，忽略同一文件中其他函数的实际调用变更
- **强制补历史上下文**：审查版本递增等孤立提交时，应主动调取历史修复内容验证一致性

### 评分通货膨胀风险
- **连续高分削弱信任**：连续出现 9/10、10/10 但实际变更微小会削弱信任
- **点赞式审查**：高评分容易导致审查偏颇，需严格执行评分校准

## 常见代码问题模式

### 配置相关问题
1. **类型安全盲区**：Python 动态类型下，配置值的运行时类型假设是高发风险点
2. **配置边界未验证**：`timeout=0`、负数、`None`、空字符串等边界值未在代码或测试中明确处理
3. **配置透传模式风险**：超时、重试等参数从配置层 → 构造函数 → 多处异步调用，需注意隐式类型假设

### 超时与重试问题
1. **硬编码超时反模式**：`page.goto`/`page.screenshot` 等浏览器操作禁止硬编码超时值
2. **双客户端超时语义不一致**：playwright 使用毫秒，httpx 使用秒，单位转换容易出错
3. **策略降级可观测性不足**：降级链的边界条件（如 `networkidle` 在 SPA 页面可能永不触发）缺乏日志记录

### 版本管理问题
1. **分离式版本号提交**：核心修复与版本递增拆分为独立 PR，容易遗漏版本一致性检查
2. **隐式版本风险**：硬性拉高最低版本可能阻断旧宿主环境用户，微小 diff 中可能隐藏生态风险
3. **版本号同步遗漏**：双版本号文件（`main.py` + `metadata.yaml`）容易不同步

### 系统解析问题
1. **脆弱字符串匹配**：解析 `/proc`、`id` 命令输出时，依赖固定数量空白符匹配
2. **容器环境假阳性**：白名单模式可避免误判（如含有 docker 字样的自定义环境变量）

## 技术模式与架构观察

### 代码模式
- **配置驱动架构**：行为完全由配置决定，灵活性高但测试组合复杂
- **单文件职责过重**：`analyzer.py` 承载 HTTP 请求、截图、策略选择等多职责，变更时需警惕牵一发动全身
- **参数构建收敛**：浏览器启动参数应统一在构建器中生成，禁止散落在重试逻辑或初始化逻辑中

### 架构观察
- **职责分散**：`analyzer.py` 同时管理 HTTP 客户端和浏览器自动化，超时配置需同时适配两种 runtime
- **错误信息演进**：从笼统错误 → 带原因提示，体现了对宿主排障体验的重视
- **防御性编程**：采用超时不阻断的设计，遇到异常时执行优雅降级

## 未来审查关注点

### 高风险领域
1. **配置边界与类型假设**：审查时须验证默认值传递路径
2. **截图功能链路**：`screenshot_only` 模式、多策略超时行为一致性、`self.timeout` 在各调用点的可见性
3. **浏览器启动参数构建**：必须收敛在单一构建器中
4. **测试覆盖要求**：涉及超时、重试、降级逻辑的 PR，必须附带或说明测试覆盖情况

### 仓库特有模式
- **双版本号文件同步**：`main.py` 与 `metadata.yaml` 必须强制同步，建议 CI 增加校验脚本
- **lifecycle 标签**：可建立 `lifecycle` 标签，专门标记安装/卸载/重置类 Issue
- **功能对称性审查**：当插件提供“安装/初始化”功能时，应主动检查是否存在对应的“卸载/清理”能力

## 优化建议

### 审查流程优化
- **强制检查历史上下文**：避免增量审查的“盲区”
- **阅读核心入口文件**：无历史记忆的增量审查若涉及新功能，须阅读核心入口文件
- **风险缓解建议**：对于删除操作，应提供失败回滚机制或手动删除指引

### 代码质量优化
- **异常包装规范**：禁止使用无上下文的笼统错误，必须传递原始异常信息
- **日志级别业务含义**：异步耗时操作的日志级别变更直接影响宿主 AstrBot 控制台排障体验
- **策略降级可观测性**：降级必须记录日志，明确包含降级原因和降级目标
