# Memory - astrbot_plugin_web_analyzer

## 常见代码问题和审查要点

- **httpx API 变更敏感区**：`proxies`(dict) → `proxy`(str) 是渐进式废弃陷阱，审查 HTTP 客户端代码时须关注 Auth、Proxy、SSL 参数的版本兼容性
- **策略模式降级路径**：`fixed/networkidle/smart` 截图策略需重点审查降级链的边界条件（如 `networkidle` 在 SPA 页面可能永不触发）和可观测性（降级日志）
- **日志级别业务含义**：异步耗时操作（如截图）的日志级别变更直接影响宿主 AstrBot 控制台排障体验，需确认是否符合生命周期（调试 vs 默认展示）
- **单文件职责过重**：`analyzer.py` 承载 HTTP 请求、截图、策略选择等多职责，变更时需警惕牵一发动全身
- **配置值类型安全**：所有使用 `self.timeout` 进行算术运算（如 `* 1000`）的地方，须确保其为 `int/float` 类型（配置解析时转换或运行时防御）
- **硬编码超时反模式**：`page.goto`/`page.screenshot` 等浏览器操作禁止硬编码超时值，必须统一使用实例超时配置
- **系统解析健壮性**：解析 `/proc`、`id` 命令输出时，严禁依赖固定数量空白符匹配，必须使用 `split()` 或正则按语义提取
- **版本号同步检查点**：版本号变更时须检查所有定义版本号的文件（`main.py`、`metadata.yaml`、`pyproject.toml`、`__init__` 等），避免遗漏
- **patch 版本上下文验证**：patch 版本递增时需确认对应的 bug 修复已合入并测试，避免空版本号提交

## 近期审查模式总结

- **小步迭代提交**：Bug 修复和新功能常拆分为多次提交，最后用单独提交修正版本约束和日志细节，增量审查易丢失全局视角
- **配置驱动架构**：行为完全由配置决定（`_conf_schema.json`），配置层→入口层→消息处理层→核心分析层
- **增量审查盲区**：仅看 diff 无法评估新功能在 `message_handler.py` 等入口文件中的调度逻辑是否正确
- **修复遮蔽效应**：当上一轮遗留问题集中在特定文件时，审查员的视线容易被锚定，忽略同一文件中其他函数的实际调用变更
- **Playwright + httpx 双客户端**：两种客户端的超时语义不同（毫秒 vs 秒），审查时须留意单位转换一致性
- **分离式版本号提交**：核心修复与版本递增常拆分为独立 PR，容易遗漏版本一致性检查，须强制回顾历史修复上下文
- **功能对称性缺失**：插件提供“安装”功能时，应主动检查是否存在对应的“卸载/清理”能力，缺失即形成 Issue 候选

## 规范建议

| 规则 | 说明 |
|------|------|
| `dep-version-context` | 依赖最低版本提升必须注明依赖的特定 API/Feature |
| `dep-compat-guard` | `quick` 策略下若涉及包管理文件变更，仍须检查依赖兼容性 |
| `schema-sync` | 配置项变更必须同步 `_conf_schema.json` |
| `strategy-logging` | 策略降级必须记录日志（含原因和降级目标） |
| `backwards-compat` | 新增配置项必须考虑旧配置文件的兼容处理 |
| `pr-desc-diff-match` | PR 描述提到的文件必须出现在 diff 中，否则追问 |
| `incremental-context` | 无历史记忆的增量审查若涉及新功能，须阅读核心入口文件 |
| `timeout-type-safety` | 超时值参与算术运算前必须确保类型为数值型 |
| `config-boundary-test` | 配置边界值（0、负数、None、空字符串）必须在代码或测试中明确处理 |
| `screenshot-timeout-consistency` | 所有浏览器操作的超时须统一使用实例超时配置，禁止硬编码 |
| `refactor-verify-equivalence` | 重构方法（如A改为调用B）必须对照审查所有调用方，确认行为绝对等价 |
| `version-bump-completeness` | 版本号变更时检查所有版本定义文件，报告列出检查结果 |
| `score-calibration` | 纯版本号/注释/格式化变更评分≤8/10；10/10 仅用于功能完成+测试覆盖+无任何建议项 |
| `delete-operation-check` | 删除关键目录前须检查进程占用、权限、失败回滚机制 |

## 经验教训

- **高评分陷阱**：改动小≠高分。功能正确但未验证边界、测试、文档时，评分应从 9/10 降至 7-8/10；纯版本号提交不超过 8/10
- **决策必须明确**：禁止使用 `unknown`，强制选择 `approve` 或 `request_changes`
- **隐式版本风险**：硬性拉高最低版本可能阻断旧宿主环境用户，微小 diff（如 +2/-2）中可能隐藏生态风险
- **错误信息丢失模式**：无上下文的异常包装（如 `Exception('失败')`）严重损害可观测性，应作为通用反模式提醒
- **环境检测假阳性陷阱**：审查容器检测时，必须逆向思考“什么情况下会被误判为 Docker”
- **评分校准**：对于包含重构但未验证等价性的 PR，评分不应超过 8/10
- **增量审查主动补历史上下文**：审查版本递增等孤立提交时，应主动调取历史修复内容验证一致性
- **异步状态同步**：`_is_installing` 与浏览器池忙碌状态的复合检查是易错点，实现中应使用锁（`asyncio.Lock`）保护

## 需要特别关注的领域

- **配置边界与类型假设**：Python 动态类型下，配置值的运行时类型假设是高发风险点，审查时须验证默认值传递路径
- **截图功能链路**：`screenshot_only` 模式、多策略超时行为一致性、`self.timeout` 在各调用点的可见性
- **浏览器启动参数构建**：必须收敛在单一构建器中，禁止在重试逻辑、初始化逻辑中出现硬编码的散落参数
- **测试覆盖要求**：涉及超时、重试、降级逻辑的 PR，必须附带或说明测试覆盖情况
- **双版本号文件同步**：`main.py` 与 `metadata.yaml` 必须强制同步，建议 CI 增加校验脚本
- **删除操作风险**：涉及删除 `.playwright` 等关键目录时，须检查进程占用、权限降级、失败回滚及手动删除指引
