# Memory - astrbot_plugin_web_analyzer

## 常见代码问题和审查要点

- **httpx API 变更敏感区**：`proxies`(dict) → `proxy`(str) 是渐进式废弃陷阱
- **策略模式降级路径**：`fixed/networkidle/smart` 截图策略需重点审查降级链边界条件
- **配置值类型安全**：所有使用 `self.timeout` 进行算术运算的地方，须确保其为 `int/float` 类型
- **硬编码超时反模式**：浏览器操作禁止硬编码超时值，必须统一使用实例超时配置
- **版本号同步检查点**：版本号变更时须检查所有定义版本号的文件，避免遗漏
- **patch 版本上下文验证**：patch 版本递增时需确认对应的 bug 修复已合入并测试

## 近期审查模式总结

- **小步迭代提交**：Bug 修复和新功能常拆分为多次提交，增量审查易丢失全局视角
- **配置驱动架构**：行为完全由配置决定，配置层→入口层→消息处理层→核心分析层
- **分离式版本号提交**：核心修复与版本递增常拆分为独立 PR，容易遗漏版本一致性检查
- **功能对称性缺失**：插件提供"安装"功能时，应主动检查是否存在对应的"卸载/清理"能力

## 规范建议

| 规则 | 说明 |
|------|------|
| `timeout-type-safety` | 超时值参与算术运算前必须确保类型为数值型 |
| `config-boundary-test` | 配置边界值（0、负数、None、空字符串）必须在代码或测试中明确处理 |
| `version-bump-completeness` | 版本号变更时检查所有版本定义文件，报告列出检查结果 |
| `score-calibration` | 纯版本号/注释/格式化变更评分≤8/10；10/10 仅用于功能完成+测试覆盖+无任何建议项 |
| `function-symmetry-review` | 当插件提供"安装/初始化"功能时，应主动检查是否存在对应的"卸载/清理"能力 |
| `async-state-synchronization` | 异步状态复合检查应使用锁保护 |
| `version-commit-integration` | 版本号变更应与核心修复合并提交，避免分离式提交导致遗漏 |

## 经验教训

- **高评分陷阱**：改动小≠高分。功能正确但未验证边界、测试、文档时，评分应从 9/10 降至 7-8/10
- **决策必须明确**：禁止使用 `unknown`，强制选择 `approve` 或 `request_changes`
- **评分校准**：纯版本号/注释/格式化变更评分≤8/10；10/10 仅用于功能完成+测试覆盖+无任何建议项
- **增量审查主动补历史上下文**：审查版本递增等孤立提交时，应主动调取历史修复内容验证一致性
- **异步状态同步**：`_is_installing` 与浏览器池忙碌状态的复合检查是易错点，实现中应使用锁保护
- **功能对称性缺失**：插件提供"安装"功能时，应主动检查是否存在对应的"卸载/清理"能力
- **版本号提交规范**：核心修复与版本递增常拆分为独立 PR，容易遗漏版本一致性检查
- **配置类型安全**：所有使用 `self.timeout` 进行算术运算的地方，须确保其为 `int/float` 类型

## 需要特别关注的领域

- **配置边界与类型假设**：Python 动态类型下，配置值的运行时类型假设是高发风险点
- **截图功能链路**：多策略超时行为一致性、`self.timeout` 在各调用点的可见性
- **测试覆盖要求**：涉及超时、重试、降级逻辑的 PR，必须附带或说明测试覆盖情况
- **双版本号文件同步**：`main.py` 与 `metadata.yaml` 必须强制同步
- **删除操作风险**：涉及删除 `.playwright` 等关键目录时，须检查进程占用、权限降级、失败回滚
