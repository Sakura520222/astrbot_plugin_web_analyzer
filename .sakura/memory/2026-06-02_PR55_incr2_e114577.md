### 2. 发现的模式

*   **代码模式**：
    *   **配置管理双轨制**：发现了一个关键模式——存在两套并行的配置访问路径。一套是面向用户配置的嵌套中文键（`“基础设置” -> “域名管理”`），用于持久化和启动加载；另一套是新Dashboard API使用的扁平化属性（`self.enable_unified_domain`），用于运行时快速读写。`_save_domain_config` 方法错误地连接了这两套轨道。这揭示了**重构或新功能集成时，对既有数据结构理解不足**的常见模式。
    *   **API安全验证层次化**：新代码在框架层面依赖了认证，但在应用层**缺少输入验证和业务逻辑校验**。这属于“纵深防御”不彻底，依赖单一安全层。这是一个需要补充的模式。
    *   **前端属性转义上下文混淆**：使用同一个函数处理HTML内容和属性值的转义，是前端安全编码中一个经典的“陷阱”。`escapeHtml` 和 `escapeAttr` 需要严格区分使用场景。

*   **架构观察**：
    *   **配置Schema是核心**：项目高度依赖 `_conf_schema.json` 来定义配置结构和UI渲染。这使得Schema成为了“单一事实来源”。**任何对配置的读、写、展示、验证操作，都必须严格遵守此Schema**。本次审查发现的路径不一致、验证缺失问题，本质上都是偏离了此原则。
    *   **插件与宿主框架的界限清晰**：Dashboard功能正确利用了AstrBot提供的`register_web_api`和Bridge SDK，保持了插件边界。但插件内部，Dashboard（管理面）与核心分析（数据面）对同一配置状态的并发访问，是架构上需要关注的潜在冲突点。

### 3. 规范完善建议

基于本次审查，建议新增或强化以下审查规则：

1.  **`config-path-consistency` (配置路径一致性)**：**审查时，必须追踪配置项的完整生命周期**（加载、读取、更新、保存）。对于任何配置读写操作，需验证其访问路径是否与Schema或主配置加载器（如`ConfigLoader`）中定义的路径严格一致。特别注意嵌套结构和扁平化访问的冲突。
2.  **`api-layered-validation` (API分层验证)**：**API端点必须实施输入验证**。即使前端校验了，后端也必须：(1) 验证数据类型与Schema定义匹配；(2) 验证值是否符合约束（如枚举、范围）；(3) 验证操作是否合法（如修改的路径是否在允许的配置子集中）。不能完全信任前端或框架的单一验证。
3.  **`html-context-sensitive-escaping` (HTML上下文敏感转义)**：在审查前端代码时，需区分 `innerHTML` (内容) 和 `setAttribute` (属性) 的上下文，强制使用对应的转义函数。建议在审查清单中加入此项。

### 4. 经验教训

1.  **功能增强必须验证与旧逻辑的兼容性**：为已有模块（如配置系统）添加新的访问方式（如Dashboard API）时，**首要任务是确保新旧路径在语义和效果上等价**。不能仅假设新代码独立正确，而必须验证其与旧逻辑的“握手”点。
2.  **Schema驱动系统，验证逻辑也必须驱动**：既然配置结构和约束由Schema定义，那么**所有动态修改配置的代码，其验证逻辑也应尽可能从Schema中动态读取**（如minimum/maximum），而非硬编码。这能保证约束的一致性。
3.  **增量审查需补充全局视角**：在增量审查中，虽然聚焦于新增代码，但对于涉及核心状态（如配置）的变更，**必须主动回溯该状态在旧代码中的使用方式和约定**，以检查兼容性。不能默认增量部分是孤立的。
4.  **安全需纵深防御**：不能因为“功能运行在受限环境”（如iframe）或“由框架认证”，就忽视应用层的安全校验。输入验证是应用自身的责任。

### 5. 增量审查反思

*   **新增提交是否解决了之前审查提出的问题？**：本次PR的提交信息（如`feat: 添加 Dashboard Web API...`）主要是在**添加新功能**，而非修复历史审查提出的具体问题（如`_safe_float`签名、内容类型默认值变更）。因此，本次审查没有发现历史Major问题的修复迹象，它们**可能仍未解决**。
*   **增量变更是否引入了新的风险？**：**是的，引入了新的风险**。主要是：
    1.  **配置管理一致性风险**（路径不匹配）。
    2.  **API安全边界风险**（缺少输入验证）。
    3.  **状态并发修改风险**（Dashboard API修改配置与核心任务可能冲突）。
*   **审查覆盖度是否因只看增量部分而有所下降？**：**是的，存在下降风险**。本次审查很好地覆盖了新增的Dashboard代码本身。但对于**新增代码与既有复杂系统（尤其是配置管理）的集成点**，审查的深度和广度可以进一步加强。例如，未系统性地将新的API更新逻辑与旧的配置加载、保存逻辑进行比对。这是增量审查的固有挑战，需要在审查时刻意地进行“集成点回溯”。

**总结**：本次审查报告质量较高，准确指出了本次增量变更中的核心问题。主要的改进空间在于：(1) 对**新旧代码集成点**的审查需要更系统化；(2) 审查时应更主动地引用和验证**项目既有的历史教训和规范**，确保新代码不重蹈覆辙或引入新变体。这有助于从“发现本次PR的问题”提升到“确保项目整体架构和代码质量持续演进”。
