## 2. 发现的模式

- **“分离式版本号提交”**：核心修复与版本递增拆分为两个独立 PR 或提交。在增量审查中容易遗漏版本一致性检查，本次审查补上了这一环。
- **版本同步检查模式**：`main.py` 与 `metadata.yaml` 双版本号，容易不同步，值得作为**固定检查点**。
- **高评分泛滥风险**：多次出现 9/10、10/10 评分，项目记忆中已警告“高评分陷阱”，本次再次出现 10/10，说明评分校准不足。

## 3. 规范完善建议

### 新增审查规则

| 规则 ID | 说明 |
|---------|------|
| `version-bump-completeness` | 版本号变更时，必须检查所有定义版本号的文件（`main.py`, `metadata.yaml`, `pyproject.toml`, `__init__`, `setup.py` 等），并在审查报告中列出检查结果。 |
| `patch-version-context` | patch 版本号递增时，需在审查摘要中确认对应的 bug 修复确实已合入且测试过（避免空版本号提交）。 |
| `score-calibration` | 纯版本号/注释/格式化变更，评分不应超过 8/10；10/10 仅可用于功能完成 + 测试覆盖 + 无任何建议项。 |

### 项目特有规范补充
- **双版本号文件**（`main.py` + `metadata.yaml`）必须作为强制同步检查点，可以在 CI 中增加脚本校验。
- **版本号提交应与变更合并**：建议要求核心修复合入时**立即**更新版本号，避免单独一次“空提交”。
