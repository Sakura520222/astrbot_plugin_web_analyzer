### 代码模式
- **双重校验模式**：配置在 `config_loader.py` 校验一次，`analyzer.py` 使用时再校验一次，体现防御性编程
- **统一构建模式**：将分散的浏览器参数构建逻辑收敛到单一方法 `_build_browser_launch_args`，符合 DRY 原则

### 架构观察
- 配置层（schema → loader）与业务层（analyzer）职责分离清晰
- 工具方法（`is_container_environment`）放在 `utils.py`，但 `analyzer.py` 采用局部导入，暗示可能存在或担忧循环依赖

## 3. 规范完善建议

### 新增审查规则
| 规则 | 说明 |
|------|------|
| **docstring 同步** | 新增/修改参数时，必须同步更新 docstring 的 Args 部分 |
| **字符串匹配健壮性** | 禁止依赖精确空白字符匹配，优先使用 `split()` 或正则 |
| **环境检测精确性** | 检测容器/运行环境时，应对变量值做精确匹配（白名单），而非仅检查非空 |

### 项目特有规范
- **沙箱配置变更**：任何涉及 `--no-sandbox` 的改动必须评估容器/非容器双场景
- **配置项生命周期**：新增配置需确认 schema 定义 → loader 校验 → 业务使用 → 文档 四处同步

## 4. 经验教训

### 值得持续关注
1. **重构时的文档债务**：代码重构容易遗漏文档更新，审查时应将 docstring diff 纳入必检项
2. **容器环境检测是坑点**：Docker/K8s/LXC/WSL/systemd-nspawn 各有特征，且内核版本差异大，检测逻辑需要持续维护
3. **安全默认值**：`sandbox_mode` 默认为 `auto` 是正确选择——安全相关配置的默认值应偏向安全而非兼容

### 技术经验沉淀
- **Linux uid_map 格式**：`/proc/self/uid_map` 的列间空格数量是内核实现细节，不应作为匹配依据，应按列解析
- **环境变量误判**：`container` 变量在 WSL2 中也可能存在，单纯检查 `os.getenv("container")` 会导致误判
- **局部导入的信号**：文件内出现局部 `from .xxx import` 通常暗示模块间耦合需要关注，审查时应追问原因而非仅建议移动位置
