# SAKURA.md - astrbot_plugin_web_analyzer

## 项目简介和技术栈
本项目是 AstrBot 的一个插件，主要用于 Web 内容分析（包含网页截图等功能）。
- **主要语言**：Python
- **核心依赖**：`httpx`（网络请求）、`playwright`（浏览器自动化）
- **双客户端架构**：同时管理 HTTP 客户端和浏览器自动化，超时配置需适配两种 runtime，语义需保持一致（注意单位差异：playwright 使用毫秒，httpx 使用秒）
- **宿主环境**：AstrBot

## 架构设计和关键决策
- **分层架构**：配置层 → 入口层 → 消息处理层 → 核心分析层。
- **配置驱动**：行为完全由配置决定（如 `self.timeout` 从配置层透传到多处异步调用），灵活性高，但也增加了测试组合的复杂度。
- **截图等待策略（策略模式）**：支持 `fixed`、`networkidle`、`smart` 三种模式，并带有降级链机制。`networkidle` 在 SPA 页面可能永远无法触发，因此降级触发条件是核心设计的关注点。
- **单文件核心倾向**：当前 `analyzer.py` 承载了较多职责（HTTP 请求、截图、策略选择等），职责分散，存在耦合。
- **防御性编程**：采用超时不阻断的设计，遇到异常时执行优雅降级。错误信息演进从笼统错误转向带原因提示，重视排障体验。
- **参数构建收敛**：浏览器启动参数（如 `--no-sandbox`）应统一在 `_build_browser_launch_args` 等构建器中生成，禁止散落在重试逻辑或初始化逻辑中。
- **版本号同步**：`main.py` 与 `metadata.yaml` 双版本号文件必须强制同步，建议在 CI 中增加脚本校验。核心修复合入时应立即更新版本号，避免单独的“空版本号提交”。

## 已知问题和注意事项
- **httpx 版本陷阱**：httpx 在 0.24 到 0.28 之间经历了 `proxies` (dict) 到 `proxy` (str) 的破坏性底层重构。最低版本硬性拉高到 0.28 可能会阻断那些使用较旧宿主环境且无法轻易升级 httpx 的用户，存在微小的生态风险。
- **SPA 页面截图局限性**：`networkidle` 策略在纯 SPA 页面可能永远无法触发，必须依赖 `smart` 策略的降级机制兜底。
- **类型安全风险**：配置值为字符串时（如 `"30"`），`self.timeout * 1000` 会引发类型错误。使用 `self.timeout` 进行算术运算处应确保其为 `int/float` 类型，在配置解析时转换或增加运行时防御。
- **配置边界未验证**：`timeout=0`、负数、`None`、空字符串等边界值未在代码或测试中明确处理，不能依赖“有默认值”假设。
- **截图超时一致性**：所有 `page.goto`/`page.screenshot` 等浏览器操作的超时必须统一使用实例超时配置，禁止硬编码。
- **错误信息丢失模式**：`Exception('截图生成失败')` 这类无上下文的异常包装会严重损害可观测性，应传递原始异常信息。
- **容器环境检测假阳性**：解析系统命令输出（如 `/proc`、`id` 命令）时，严禁依赖固定数量的空白符进行匹配，应使用 `split()` 或正则按语义提取；同时注意白名单模式可避免误判（如含有 docker 字样的自定义环境变量）。
- **版本号提交风险**：纯版本号递增提交本身风险低，但需确认所有版本定义文件已同步，且对应的 bug 修复确实已合入并测试过。

## 审查中发现的重要模式
- **小步迭代提交**：针对一个 Bug 修复和一个新功能，常拆分成多次提交，最后用单独的提交修正版本约束和日志细节。
- **增量审查盲区与修复遮蔽效应**：仅看 diff 极易忽略依赖版本提升的连锁反应及整体功能的闭环。当审查修复验证时，必须强制跳出 diff 标红区域，通读涉及文件的完整上下文，验证重构等价性（如“重构 A 方法改为调用新方法 B”时，需对照审查所有调用方确认行为绝对等价）。
- **光环效应与点赞式审查**：高评分容易导致审查偏颇。改动小 ≠ 高分，功能正确性 ≠ 完备性（测试、边界、文档）。对于包含重构的提交，若未验证重构等价性，评分不应超过 8/10。纯版本号/注释/格式化变更，评分不应超过 8/10。
- **日志级别的业务含义**：在耗时较长的异步操作中，日志级别（INFO 与 DEBUG）的变更直接影响宿主控制台的排障体验，审查时需关注其是否符合生命周期。
- **配置透传模式**：超时、重试等参数从配置层 → 构造函数 → 多处异步调用，属于“配置驱动”典型实践，但需注意隐式类型假设风险。
- **系统解析逻辑演进**：从脆弱的字符串精确匹配向结构化语义解析转变，引入白名单机制收敛枚举边界。
- **分离式版本号提交模式**：核心修复与版本递增拆分为两个独立 PR 或提交时，增量审查中容易遗漏版本一致性检查，需强制检查所有版本定义文件。
- **评分通货膨胀风险**：连续出现 9/10、10/10 但实际变更微小会削弱信任，需严格执行评分校准。

## 团队约定和规范
- **依赖版本提升规范**：提升最低版本号时，提交信息或代码注释必须明确指出依赖该特定版本的 API 或 Feature。
- **审查策略降级限制**：对于 `quick` 审查策略，若变更文件包含包管理文件（如依赖清单），即使代码行数极少，也必须触发“依赖兼容性”基础检查。
- **配置同步规则 (schema-sync)**：配置项变更必须同步更新 `_conf_schema.json`。
- **策略可观测性 (strategy-logging)**：策略降级必须记录日志，明确包含降级原因和降级目标。
- **向后兼容规则 (backwards-compat)**：新增配置项必须考虑旧配置文件升级时的兼容处理（提供合理的默认值）。
- **PR 描述一致性**：PR 描述中提到的修改文件必须出现在 diff 中，否则需在审查中追问原因。
- **审查决策明确化**：不允许使用模糊决策（如 `unknown`），必须强制选择 `approve` 或 `request_changes`。
- **超时类型安全 (timeout-type-safety)**：所有使用 `self.timeout` 进行算术运算的地方，须确保其为 `int/float` 类型。
- **配置边界测试 (config-boundary-test)**：配置边界值（0、负数、None、空字符串）必须在代码或测试中明确处理。
- **截图超时一致性 (screenshot-timeout-consistency)**：所有浏览器操作的超时须统一使用实例超时配置，禁止硬编码。
- **PR 测试要求 (pr-test-requirement)**：涉及超时、重试、降级逻辑的 PR，必须附带或说明测试覆盖情况。
- **系统解析健壮性 (system-parse-robustness)**：解析系统命令输出的代码，严禁依赖固定数量的空白符进行匹配，必须使用 `split()` 或正则按语义提取。
- **重构等价性验证 (refactor-verify-equivalence)**：当变更中包含“重构 A 方法，改为调用新方法 B”时，必须对照审查所有调用方，确认重构前后行为绝对等价。
- **版本号变更完整性 (version-bump-completeness)**：版本号变更时，必须检查所有定义版本号的文件（`main.py`, `metadata.yaml`, `pyproject.toml`, `__init__`, `setup.py` 等），并在审查报告中列出检查结果。
- **补丁版本上下文验证 (patch-version-context)**：patch 版本号递增时，需在审查摘要中确认对应的 bug 修复确实已合入且测试过。
- **评分校准 (score-calibration)**：纯版本号/注释/格式化变更，评分不应超过 8/10；10/10 仅可用于功能完成 + 测试覆盖 + 无任何建议项。
