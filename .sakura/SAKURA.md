# SAKURA.md - astrbot_plugin_web_analyzer

## 项目简介和技术栈
本项目是 AstrBot 的一个插件，主要用于 Web 内容分析（包含网页截图等功能）。
- **主要语言**：Python
- **核心依赖**：`httpx`（网络请求），对 httpx 的版本有严格底线要求（当前最低版本 >=0.28.0，因为依赖了 `proxy` 参数的稳定化 API）。
- **宿主环境**：AstrBot

## 架构设计和关键决策
- **分层架构**：配置层 → 入口层 → 消息处理层 → 核心分析层。
- **配置驱动**：行为完全由配置决定，灵活性高，但也增加了测试组合的复杂度。
- **截图等待策略（策略模式）**：支持 `fixed`、`networkidle`、`smart` 三种模式，并带有降级链机制（如 `smart` 在 `networkidle` 超时后触发降级）。`networkidle` 在 SPA 页面可能永远无法触发，因此降级触发条件是核心设计的关注点。
- **单文件核心倾向**：当前 `analyzer.py` 承载了较多职责（HTTP 请求、截图、策略选择等），存在一定的耦合。
- **防御性编程**：采用超时不阻断的设计，遇到异常时执行优雅降级。

## 已知问题和注意事项
- **httpx 版本陷阱**：httpx 在 0.24 到 0.28 之间经历了 `proxies` (dict) 到 `proxy` (str) 的破坏性底层重构。最低版本硬性拉高到 0.28 可能会阻断那些使用较旧宿主环境且无法轻易升级 httpx 的用户，存在微小的生态风险。
- **SPA 页面截图局限性**：`networkidle` 策略在纯 SPA 页面可能永远无法触发，必须依赖 `smart` 策略的降级机制兜底。
- **代理配置边界**：网络请求的代理参数变更需严格验证 `None` 和空字符串等边界情况。

## 审查中发现的重要模式
- **小步迭代提交**：针对一个 Bug 修复和一个新功能，常拆分成多次提交，最后用单独的提交修正版本约束和日志细节。
- **增量审查盲区**：仅看 diff 极易忽略依赖版本提升的连锁反应及 PR 整体功能的闭环情况。对于涉及新功能的增量审查，必须强制阅读核心入口文件（如 `main.py`）以理解全貌，而非仅盯着 diff。
- **光环效应与点赞式审查**：高评分容易导致审查偏颇，缺乏对复杂度不同模块的区分和深度质询。简单的参数替换不应与核心策略变更获得同等赞誉。
- **日志级别的业务含义**：在耗时较长的异步操作（如截图策略）中，日志级别（INFO 与 DEBUG）的变更直接影响宿主控制台的排障体验，审查时需关注其是否符合生命周期。

## 团队约定和规范
- **依赖版本提升规范**：提升最低版本号时，提交信息或代码注释必须明确指出依赖该特定版本的 API 或 Feature（如为何需要 httpx>=0.28）。
- **审查策略降级限制**：对于 `quick` 审查策略，若变更文件包含包管理文件（如依赖清单），即使代码行数极少，也必须触发“依赖兼容性”基础检查，不能完全跳过。
- **配置同步规则 (schema-sync)**：配置项变更必须同步更新 `_conf_schema.json`。
- **策略可观测性 (strategy-logging)**：策略降级必须记录日志，明确包含降级原因和降级目标。
- **向后兼容规则 (backwards-compat)**：新增配置项必须考虑旧配置文件升级时的兼容处理（如提供合理的默认值）。
- **PR 描述一致性**：PR 描述中提到的修改文件必须出现在 diff 中，否则需在审查中追问原因。
- **审查决策明确化**：不允许使用模糊决策（如 `unknown`），必须强制选择 `approve` 或 `request_changes`。
