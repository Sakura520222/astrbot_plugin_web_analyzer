# 审查规则与编码规范

## 审查决策规则

### 评分校准规则
- **纯版本号/注释/格式化变更**：评分不应超过 8/10
- **功能完成但未验证边界、测试、文档**：评分应从 9/10 降至 7-8/10
- **10/10 评分标准**：仅用于功能完成 + 测试覆盖 + 无任何建议项
- **重构等价性验证**：若包含重构但未验证等价性，评分不应超过 8/10

### 决策明确化规则
- **禁止使用模糊决策**：不允许使用 `unknown`，必须强制选择 `approve` 或 `request_changes`
- **增量审查决策**：即使只看增量 diff，也必须基于完整上下文做出明确决策

## 代码审查要点

### 配置相关规则
| 规则 ID | 说明 |
|---------|------|
| `schema-sync` | 配置项变更必须同步更新 `_conf_schema.json` |
| `backwards-compat` | 新增配置项必须考虑旧配置文件的兼容处理（提供合理的默认值） |
| `config-boundary-test` | 配置边界值（0、负数、None、空字符串）必须在代码或测试中明确处理 |
| `timeout-type-safety` | 所有使用 `self.timeout` 进行算术运算的地方，须确保其为 `int/float` 类型 |

### 超时与重试规则
| 规则 ID | 说明 |
|---------|------|
| `screenshot-timeout-consistency` | 所有浏览器操作的超时须统一使用实例超时配置，禁止硬编码 |
| `pr-test-requirement` | 涉及超时、重试、降级逻辑的 PR，必须附带或说明测试覆盖情况 |
| `strategy-logging` | 策略降级必须记录日志，明确包含降级原因和降级目标 |

### 版本管理规则
| 规则 ID | 说明 |
|---------|------|
| `version-bump-completeness` | 版本号变更时检查所有版本定义文件（main.py, metadata.yaml, pyproject.toml 等） |
| `patch-version-context` | patch 版本号递增时需确认对应的 bug 修复已合入并测试 |
| `dep-version-context` | 依赖最低版本提升必须注明依赖的特定 API/Feature |
| `dep-compat-guard` | `quick` 策略下若涉及包管理文件变更，仍须检查依赖兼容性 |

### 代码质量规则
| 规则 ID | 说明 |
|---------|------|
| `refactor-verify-equivalence` | 重构方法必须对照审查所有调用方，确认行为绝对等价 |
| `system-parse-robustness` | 解析系统命令输出时，严禁依赖固定数量空白符匹配，必须使用 split() 或正则按语义提取 |
| `pr-desc-diff-match` | PR 描述提到的文件必须出现在 diff 中，否则追问 |

## 项目特定规范

### 双客户端架构注意事项
- **超时语义一致性**：playwright 使用毫秒，httpx 使用秒，审查时须留意单位转换
- **配置透传模式**：超时、重试等参数从配置层 → 构造函数 → 多处异步调用，需注意隐式类型假设风险

### 截图功能审查要点
- **策略降级链**：`fixed`、`networkidle`、`smart` 三种模式的降级触发条件是核心关注点
- **SPA 页面局限性**：`networkidle` 策略在纯 SPA 页面可能永远无法触发，必须依赖 `smart` 策略的降级机制兜底
- **错误信息可观测性**：异常包装必须传递原始异常信息，禁止使用无上下文的笼统错误

### 浏览器启动参数规则
- **参数构建收敛**：浏览器启动参数（如 `--no-sandbox`）应统一在 `_build_browser_launch_args` 等构建器中生成
- **禁止散落参数**：禁止在重试逻辑或初始化逻辑中出现硬编码的散落参数

## 审查流程规范

### 增量审查注意事项
1. **强制检查历史上下文**：审查版本递增等孤立提交时，应主动调取历史修复内容验证一致性
2. **避免修复遮蔽效应**：当上一轮遗留问题集中在特定文件时，审查员需警惕视线被锚定，忽略同一文件中其他函数的实际调用变更
3. **阅读核心入口文件**：无历史记忆的增量审查若涉及新功能，须阅读核心入口文件（如 `message_handler.py`）
4. **PR描述与实际变更对照**：PR描述中提到的文件必须出现在diff中，否则需追问原因

### 评分通货膨胀防范
- **高评分陷阱**：改动小 ≠ 高分，功能正确性 ≠ 完备性
- **连续高分审查**：连续出现 9/10、10/10 但实际变更微小会削弱信任，需严格执行评分校准
- **评分校准规则**：
  - 纯版本号/注释/格式化变更：评分不应超过 8/10
  - 功能完成但未验证边界、测试、文档：评分应从 9/10 降至 7-8/10
  - 10/10 评分标准：仅用于功能完成 + 测试覆盖 + 无任何建议项
  - 重构等价性验证：若包含重构但未验证等价性，评分不应超过 8/10

### 审查质量评估维度
- **覆盖度**：检查是否遗漏关键文件、边界条件、兼容性考虑
- **准确度**：避免"光环效应"，对复杂度不同的变更应有差异化评价
- **完整性**：避免表面化审查，需验证关键路径和可观测性

## 环境检测与兼容性

### 容器环境检测
- **假阳性陷阱**：审查容器检测时，必须逆向思考“什么情况下会被误判为 Docker”
- **白名单机制**：引入白名单机制收敛枚举边界，避免误判

### 依赖兼容性
- **httpx 版本陷阱**：httpx 在 0.24 到 0.28 之间经历了 `proxies` (dict) 到 `proxy` (str) 的破坏性底层重构
- **生态风险**：最低版本硬性拉高到 0.28 可能会阻断那些使用较旧宿主环境且无法轻易升级 httpx 的用户
